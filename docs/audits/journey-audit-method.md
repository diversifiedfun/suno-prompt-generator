# Journey Audit — portable (browser extension)

You run a **repeatable, code-grounded UX/UI journey audit** against the extension repo you're checked out in, across **every surface (side panel / popup / options page) and every content-script → host-page handoff**, and file a GitHub issue per **new** finding so a human can triage later. You do **not** fix anything, edit extension code, or open PRs. The only writes are `gh issue` / `gh label`.

This is the extension successor to the manual journey audits in `docs/audits/` (where they exist). If this repo has a `docs/audits/` folder, **read the latest audit first** — it sets the bar, the voice, and the findings that already exist.

**An extension has no routes and no live URL.** "Journeys" are flows across surfaces + tabs, and the make-or-break moments are the **message seams** (panel ↔ background ↔ content script) and the **content-script → third-party-DOM handoff**. Hunt those.

---

## STEP 0 — No-change skip gate (do this before anything else)

Runs are stateless, so the last-audited commit lives in a **pinned state issue**.

1. Current commit: `HEAD_SHA=$(git rev-parse HEAD)` and `git log -1 --format='%h %s'`.
2. Find the state issue (search **all** states):
   `gh issue list --label journey-audit --state all --search '[journey-audit-state] in:title' --json number,title,body,state --limit 5`
   Its title is exactly `[journey-audit-state] audit bookkeeping — do not close`.
3. Decide skip vs run — **fail safe: when in doubt, run a full audit. Never silently skip.**
   - State issue **missing** → first run → full audit.
   - State issue exists but **closed** → treat as missing: reopen (`gh issue reopen <num>`), note it, run a **full audit**.
   - State issue open but `last_audited_sha:` **absent/unparseable** (not a 40-hex sha) → full audit.
   - `last_audited_sha == HEAD_SHA` → **SKIP THE ENTIRE RUN.** Print `No code changes since <sha> — skipping.` and exit 0.
   - Else `git diff --name-only <last_audited_sha> HEAD`: if **every** changed path is docs-only (`docs/**`, `**/*.md`, `.claude/**`, `.github/**`) and **no** extension source changed (the surface HTML/CSS/JS, content scripts, background, `manifest.json`) → **SKIP** (`Only non-source changes — skipping.`) and exit 0. NOTE: `manifest.json` IS source here (permissions/host-matches live there) — a manifest-only change is NOT a docs skip.
4. **Known limitation (state it in the summary):** this gate detects *committed code* changes only. Chrome-version / host-site (third-party DOM) changes are **not** detected — and a host redesign is the single most common thing that silently breaks an extension. Accepted, since the audit is code-grounded, but call it out. Never auto-close the state issue.

## Method (non-negotiable)

1. **Code-grounded, every claim cites `file:line`.** No speculation. Trace real control flow — tab switchers, conditional renders, message handlers, `chrome.storage` reads, breakpoint/panel-width CSS.
2. **Journeys, not screens.** Well-built panels leak at the *handoffs* — the message boundary, the host-page injection, the storage round-trip. Hunt the doorways, not the rooms.
3. **Cross the boundary.** The defining risk of an extension is code running in TWO worlds (the panel's page and the host site's page) talking over `chrome.runtime`/`chrome.tabs` messaging. Trace each message from sender → listener → response, and every `chrome.scripting.executeScript` injection. A dropped/duplicated/dead listener is a top-severity bug.
4. **Parallelize.** Use the Task tool to fan out journeys as concurrent sub-audits, then merge. Each sub-auditor cites `file:line`.
5. **Audit the degraded paths as first-class:** no API key / offline, host tab not present, host user not logged in, host page in the wrong mode, extension just reloaded (orphaned content script), storage at quota. These are where real users live.

## This project's journeys (it differs per repo)

**If `docs/audits/journey-map.md` exists, it is authoritative.** It was generated by `/journey-audit-init-extension` and names this extension's real surfaces, tabs, content-script host matches, message seams, credentials + storage model, and known journeys. Use it as the journey list; fall back to code-discovery only to fill gaps or verify a claim is current. If absent, discover from the repo:
- **Manifest:** read `manifest.json` — UI surfaces (`action.default_popup`, `side_panel.default_path`, `options_page`/`options_ui`), `background.service_worker`, `content_scripts[].matches` (host sites), `permissions` / `host_permissions` / `optional_permissions`, `commands`, `web_accessible_resources`.
- **Tabs:** enumerate in-panel tabs/views (grep HTML for `data-tab` / `role="tab"` / `.tab-panel` + the JS switcher). Tab → handler.
- **Message seams:** `grep -rE "chrome\.(runtime\.sendMessage|runtime\.onMessage|tabs\.sendMessage|scripting\.executeScript)"` — map every hop.
- **Host coupling:** which content script touches which third-party DOM, and where its selectors live.

Map each flow to one of these **nine extension archetypes** and trace it end-to-end. **"User" includes the extension user AND, where relevant, the developer generating a key.** Note which don't apply:

1. **Install → first value** — CWS install / load-unpacked → open the surface (side panel via the action icon, or popup) → the FIRST real output. Watch: MV3 service-worker cold-start delays, a first-run state that's blank or broken, a surface that does nothing until setup with no explanation, whether ANY value is reachable before credentials.
2. **Permissions & trust** — the install-time prompt (from `permissions`/`host_permissions`) and any RUNTIME prompt/banner. Watch: an alarming permission (`debugger` → Chrome's "started debugging this browser" banner; `<all_urls>`; `tabs`) with **no in-UI rationale** so the user thinks it's malware; an `optional_permissions` request fired with no lead-up; over-broad host matches. Trust dies at an unexplained scary prompt.
3. **Credentials / API-key onboarding** — the user supplies their own key/token (or OAuth). Watch: **untrimmed key** stored verbatim → a valid key silently 401s (clipboard whitespace); no "where do I get one" link at the field; no format check or "test key"; a key shown once and lost; stored in `.sync` (roams, leak surface) vs `.local`; key echoed to logs.
4. **Core action → host-page handoff (THE defining seam)** — panel generates something → content script fills/reads the host site's DOM. Watch, exhaustively: host tab not open; host user **not logged in** (and a fill that "succeeds" against a logged-out widget); host page in the wrong mode/route so the target field is absent; **stale selectors** after a host redesign (does it fail LOUDLY or silently no-op?); a **dead content-script listener after an extension reload** (a one-shot boolean install guard is the bug; a handler-ref swap is the fix); **captcha / anti-bot** walls after rapid actions; a CDP/`debugger` drive that doesn't verify it landed. Every failure needs a message that says exactly what to do next.
5. **Persistence & return** — `chrome.storage` round-trips + cross-session/cross-window restore. Watch: state not restored on reopen; a "load"/"restore" affordance that updates the DOM but **not** the persisted state (lost on reopen); `.local` quota overflow or a cap that silently evicts; a concurrent-write race (one-array-append vs per-key); data wiped on extension update.
6. **Discovery & navigation** — the tabs/views are this world's nav. Watch: unclear/overloaded tab labels (the same verb across multiple tabs), a feature buried in a collapsed accordion the user never opens, no info scent for which tab does what, a data list styled like help text so users don't recognize it as their own content.
7. **Help & error recovery** — how every failure routes the user somewhere useful. Watch: a **bare `alert()`/`confirm()`** where inline status is the norm; a silent `catch{}` that swallows a save/paste failure (dead click); an error with no next step; no fallback when the model call or the host handoff fails; help copy that doesn't match the shipped UI.
8. **Offboarding & data hygiene** — clear stored data, revoke/replace the key, export, and what uninstall leaves behind. Watch: no "clear my data"; a key that can't be removed; a "clear all" that also deletes protected/starred items (contradicting a "kept forever" promise); orphaned `chrome.storage` after uninstall where the extension promised otherwise; a disconnect that doesn't actually disconnect.
9. **Share / export** — the artifacts the tool produces packaged to travel. Watch: outputs that are copy-only with no export/download/shareable file; a multi-step manual copy where one action should do it; no way to hand a result to someone else.

## Cross-cutting lenses (apply to EVERY journey above)

Check these at each step of each journey; file findings under the journey/step where the lens fails:
- **Empty & first-run states** — every list/result/history with no data shows guidance + a next action, never a blank or raw error. First-time vs returning states differ.
- **Error states, recovery & microcopy** — every failure (no host tab, not logged in, host DOM changed, network/API, quota, validation) has a clear, human, *recoverable* message. No bare `alert()`. Never rely on color alone.
- **Accessibility (narrow-panel heuristic pass)** — labels on inputs, accessible names on icon-only buttons (`title`/`aria-label`), `aria-live` on status/toast regions so paste/save feedback is announced, visible keyboard focus (`:focus-visible`), sufficient contrast on faint secondary text, `prefers-reduced-motion`.
- **Perceived performance** — click→feedback on every action; no dead-feeling clicks; loading state during the model call and the host handoff.
- **Third-party-DOM resilience** — the content script that couples to the host site: is the DOM coupling isolated to one file, and does it **fail loudly** (surface a toast the user can act on) rather than silently no-op when the host redesigns? A silent no-op here is the classic "extension just stopped working."
- **MV3 service-worker lifecycle** — a message sent to a sleeping/terminated SW or an orphaned content script (post-reload); does the code re-inject/retry or dead-click?
- **AI-generation reliability** (if it calls a model) — timeout / partial / empty / garbage output, retry/regenerate present, quota/credit exhaustion handled (vs infinite spinner or silent offline-fallback that masquerades as success), and a "did it actually work" confirmation.

## Live spot-check (best-effort, degrade gracefully)

> **Where the live pass runs (by design):** the **scheduled cloud routine has no browser** — it will record `live-pass: skipped (no browser)` every run, and that is expected. The authoritative live pass is a **separate local run**: load the extension unpacked in Chrome (`chrome://extensions` → Developer mode → Load unpacked → the extension dir) and drive the surface with claude-in-chrome. There is **no URL to hit** — an extension surface is `chrome-extension://<id>/…`, only reachable in a browser with the extension installed.

Code-grounded findings are the **source of truth**. Additionally, in a local run only, try a live pass:
1. Confirm a browser + the claude-in-chrome tools are available. If not → skip.
2. Load unpacked, open the side panel / popup, and walk the top 1–2 journeys, confirming what the code-trace predicted (e.g. "the paste really fails until the host is in Custom mode"). For the host handoff, the host site (e.g. suno.com) must be open + logged in.
3. If no browser / can't load / it flakes → note `live-pass: skipped (<reason>)` and proceed on code-grounded findings alone. **Never block or fail the run on the live pass.** Never fabricate a live observation.

**Honesty contract (mandatory):** the run summary AND the state-issue run-log MUST record live-pass status explicitly — either `live-pass: ran — loaded unpacked, walked <journeys>` or `live-pass: skipped (<reason>)`. Each finding's "Live spot-check" field must say exactly one of: a real observation you made, or `not run (code-grounded only)`. A finding derived from code analysis is fine — but do NOT phrase it as if you watched it in a browser unless you did.

## Severity rubric

- **Critical** — silent failure on the core action or the host handoff; an on-screen promise the code contradicts (e.g. "kept forever" that Clear-all deletes); a trust-breaking permission/data issue; a dead-end on the first-run path.
- **High** — degraded experience for the majority (a whole surface/tab broken, the paste seam failing on a common host state), or a trust-eroding inconsistency.
- **Medium** — polish, honesty, missing states that hurt but don't block.
- **Low** — cosmetic / affordance gaps.

## Dedup protocol (BEFORE filing anything)

Derive the fingerprint **mechanically from the most stable identifiers**, never from prose:

> **`JAX-<archetype#>-<primary-file-basename>-<primary-symbol>`** — lowercase kebab, no extension, no path, no line number. (`JAX` = journey-audit-extension, distinct from the web sibling's `JA-`.)
> - *primary file* = the single file most central to the bug (basename only, e.g. `suno-fill.js` → `suno-fill`).
> - *primary symbol* = the function / handler / message-type / tab the bug lives in (e.g. `sunoFillHandler` → `sunofillhandler`), NOT a description of the symptom.
> - Examples: `JAX-4-suno-fill-sunofillhandler`, `JAX-3-storage-setsettings`, `JAX-8-gen-history-cleargenerations`.
> Two runs that find the same bug land on the same file+symbol → same fingerprint. If you can't name a stable symbol, use the tab id or content-script file — never the symptom.

1. `gh issue list --label journey-audit --state open --limit 300 --json number,title`
2. Titles start with `[<fingerprint>]`. **Match on the fingerprint prefix only**, ignoring the human-readable tail.
3. Fingerprint already has an open issue → **skip it** (no re-file, no comment).
4. A previously-filed finding now looks **fixed** in code → do NOT close; list under "Likely resolved" for a human.
5. Collapse near-identical findings within one run (no dup fingerprints in a run).

## Verify before you file (adversarial pass — do NOT skip)

A wrong finding costs human triage time. Before filing each candidate:
1. **Re-open the cited `file:line` and confirm the code STILL does what the finding claims, on this checkout.** If a guard/fix you missed exists nearby → **DROP it** (or move to "Likely resolved").
2. **Argue the other side in one sentence:** why might this NOT be a bug (intentional, handled elsewhere, dead code, already-shipped fix)? If you can't refute it with a specific code reference → **DROP it.**
3. **Prefer dropping a shaky finding over filing a false positive.** Report how many you dropped (and why) — that number is a quality signal.

"Code-grounded" means *re-confirmed at file time*, not "looked plausible during the trace."

## Filing format

Ensure labels exist (idempotent):
```
gh label create journey-audit --color BFD4F2 --description "Findings from the scheduled UX journey audit agent" 2>/dev/null || true
gh label create needs-triage  --color FBCA04 --description "Filed by an automated agent; awaiting human triage" 2>/dev/null || true
```
Per NEW finding:
```
gh issue create --title "[<fingerprint>] <one-line problem>" \
  --label journey-audit --label needs-triage --body "<body>"
```
Body template:
```
**Journey:** <archetype# + name>
**Surface:** side panel | popup | options | content-script@<host>
**Severity:** Critical | High | Medium | Low
**Fingerprint:** <JAX-...>

### What breaks
<the friction/bug in journey terms — what the user experiences>

### Evidence (code-grounded)
- `path/to/file.js:LINE` — <what this does that causes it>
- `path/to/other.js:LINE` — <corroborating ref>

### Live spot-check
<loaded unpacked, walked X — or "not run (code-grounded only)">

### Affected path
First-run / no-key | Host-not-logged-in | Host-wrong-mode | Post-reload | Normal

### Suggested fix (smallest)
<cheapest change that closes it, 1–3 sentences>

---
🤖 Filed by the scheduled journey-audit agent. Verify against current code before acting.
```

## STEP FINAL — Update the state issue

After filing (even if zero), record what you audited so the next run can skip:
- State issue exists → `gh issue edit <num> --body "..."`.
- Else → `gh issue create --title "[journey-audit-state] audit bookkeeping — do not close" --label journey-audit --body "..."`.

State body shape:
```
Do not close — used by the scheduled journey-audit agent to skip unchanged runs.

last_audited_sha: <full HEAD sha>
last_run_utc: <date -u>
last_run_filed: <N new issues>  skipped_existing: <M>

### Run log (most recent first)
- <date> · <sha7> · filed N, skipped M · surfaces <panel/popup/options> · live-pass: <ran|skipped reason>
```

## Guardrails

- **Verify you can file before you start.** `gh auth status` + a no-op `gh issue list`. If issue creation isn't available, STOP and print a clear failure — never fabricate "filed".
- **Read-only on code.** No edits, commits, or PRs. Writes = `gh issue` / `gh label` only.
- **Cap noise: ≤12 issues per run.** If more, file the Critical/High ones and list the rest in the summary.
- Works for any of Molly's Chrome/MV3 extensions; never hardcode one extension's surfaces — discover them from the repo in hand.

## Run summary (always end with this)

- SKIP banner if STEP 0 short-circuited (sha + reason), otherwise:
- Findings by severity + surface (counts)
- New issues filed (fingerprint + # + URL)
- Skipped as already-open (fingerprint + #)
- **Candidates dropped in the verify pass** (count + one-line reason each — a run that drops 0 of many is suspicious)
- "Likely resolved since last run" (human verifies — NOT auto-closed)
- **Per-archetype coverage accounting (mandatory — every applicable archetype gets a line).** For each of the 9 (skip only genuinely N/A, and say why): `archetype N — <filed K> | <clean: checked X, Y, no issue> | N/A (<reason>)`. A bare "0 findings" is NOT acceptable — "clean" must name what you traced, so a reader can tell *audited-and-clean* from *never-looked*. Pay special attention to archetype 4 (host handoff) — a clean pass there is suspicious; name the exact host states you checked (not-logged-in, wrong-mode, post-reload).
- **live-pass: ran (surfaces walked) | skipped (reason)** — explicit, per the honesty contract
- State-issue bookkeeping: existed / reopened / new sha recorded
- Anything you couldn't trace / needs a human

**Post this whole summary as a comment on the `[journey-audit-state]` issue** (in addition to printing it), so every run's coverage + drop-count + live-pass status is auditable from GitHub.
